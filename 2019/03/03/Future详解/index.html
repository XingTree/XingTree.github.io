Future详解

发表于 2019-03-03 | 分类于 Java基础

run方法

看到这,可能有点疑惑,没关系,我们先有个大概的印象,既然FutureTask要被当做Runnable去执行,那肯定是有run方法的,我们先来看看FutureTask的run方法:

首先判断当前任务的状态是否为NEW,若不为NEW,则说明该任务已经执行结束,或者已经被取消了,就不再执行,然后通过CAS替换当前任务中runner属性为当前的线程,若替换不成功,则不执行(防止并发操作).

之后再次判断state值是否为NEW(双重判断),然后调用Callable的call方法,若正常执行,则调用set()方法设置结果,若调用失败,则调用setException()方法设置异常.

接着再来看set()方法和setException()方法:

我们可以看到,这两个方法的过程十分相似,首先都是将状态先通过CAS设置为COMPLETING,然后将结果(或异常)设置到outcome中,然后set方法将状态设置为NORMAL,setException方法是将状态设置为EXCEPTIONAL,最终都是调用了finishCompletion()方法.finishCompletion方法就是唤醒阻塞在获取当前任务结果上的线程唤醒(因为任务已经到达了最终状态),之后在分析.

我们可以看出,任务的状态在run方法中有两条分支:

正常执行获得结果:NEW -> COMPLETING -> NORMAL

执行发生异常:NEW -> COMPLETING -> EXCEPTIONAL

cancel方法

但是我们在之前看到,任务的状态还有取消的状态,这个我们就需要来看cancel(boolean)方法的代码了:

cancel方法需要我们传一个boolean类型的参数,这个参数若为ture,表示会中断任务,若参数为false,则不会中断该任务.

cancel方法首先判断任务状态是否为NEW,不为NEW,则直接返回false,任务取消失败.若状态为NEW,则使用CAS将状态替换掉,若传的参数为true,则状态替换为INTERRUPTING,若传的参数为false,则状态替换为CANCELLED.

之后根据传的参数,判断是否需要中断线程,若传的参数为true,则中断线程,中断线程后,将状态设置为INTERRUPTED.

最后,都执行finishCompletion()方法唤醒其他线程.

我们可以看出,任务取消的状态也有两条分支:

不中断线程的状态变化:NEW -> CANCELLED

中断线程的状态变化:NEW -> INTERRUPTING -> INTERRUPTED

get方法

到这里,我们讲完了一个任务状态所有的变化,我们可以看到,每当任务状态变为最终状态时(变为该状态后就不会改变了),都会去执行finishCompletion()方法,这个方法是用来唤醒阻塞在get上的其他线程的,所以我们先来看get方法:

get方法分为一直阻塞到任务结束和只阻塞一定的时间的两种.

我们可以从上述代码中看到,两种都是要判断当前状态是否小于等于COMPLETING,因为大于COMPLETING的话,表示该任务已经执行结束了,若任务未结束,则调用awaitDone方法阻塞,若任务已经结束了,则直接调用report方法返回结果.

我们先来看看awaitDone方法:

awaitDone方法有两个参数,第一参数标识是否有超时时间,第二个参数是设置的超时时间.

具体流程不做分析,流程很简单,就是若任务未完成,则构造一个WaitNode节点,然后将FutureTask中的waiters属性指向该节点,再将该节点的next指向之前的WaitNode,组成一个单向链表结构,这个单向链表的结构,就是所有阻塞在get时的线程.

然后就分为是否有超时时间两种,若无超时时间,则调用LockSupport.park(this)一直阻塞,若有超时时间,则调用LockSupport.parkNanos(this, nanos)阻塞一段时间,若超过超时时间,则调用removeWaiter方法将当前线程所在的节点删除.

当任务被完成时(状态变为最终状态,调用finishCompletion方法唤醒阻塞线程),线程被唤醒,最终awaitDone方法返回任务的状态.

之后就是调用report方法返回结果:

若正常完成,则返回正常结果.

若任务被取消,则抛出CancellationException异常.

若任务出现异常,则抛出封装了抛出的异常的ExecutionException.

该方法也很简单,就是遍历之前我们所说的WaitNode链表结构,唤醒所有阻塞的线程.
