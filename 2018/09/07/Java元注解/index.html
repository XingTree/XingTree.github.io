发表于 2018-09-07 | 分类于 Java基础

Java注解

Java注解是JDK 1.5引入的新特性,我理解的Java注解本身只是一个标记,注解本身并没有实际的作用,要发挥作用都是需要额外的地方取得这个标记,然后再去做一些其他事情.

就像我们经常在类中使用的@Override注解,我们都知道这个注解是在方法上定义,表示这个方法是一个继承于父类的重写方法.当我们用@Override标记的方法与在父类中的方法参数不符或者在父类根本就不存在这个方法时,编译就会报错.但是如果我们直接把@Override这个标记去掉,则即使该方法与父类中方法参数不符或父类不存在该方法,编译也不会报错.

在这里,@Override其实就起到一个标记的作用,标记被修饰的方法是父类的重写方法,当编译器去编译代码时,检测到一个方法被@Override修饰,就会去查找父类的方法,然后判断该方法是否是合法的重写方法.所以这里我们可以看到实际上@Override注解只是起到一个标记作用,真正去使用,去判断逻辑的是编译器.

@Override是JDK自带的注解,当然,Java也允许我们自定义注解进行使用:

这样定义一个注解后,我们就可以在代码中使用@TestAnnotation来使用注解了.

Java元注解

当我们可以自定义注解后,你可能会有这样的疑问,为什么@Override只能修饰方法,不能修饰类或者属性呢?为什么@Override注解编译之后的字节码反编译之后就没有了呢?(读者可以自己去证实一下.)

这些都是由JDK中提供的元注解来定义的,元注解可以理解为定义注解的注解,我们可以看看@Override的源码:

我们可以看到@Override注解的定义中,也被其他注解修饰了,这些就是JDK的元注解.

@Retention

@Retention主要定义了注解的作用范围,我们可以看该注解的源码:

该注解中有个RetentionPolicy枚举,就是定义的注解的作用范围,有三种范围:

  1. RetentionPolicy.SOURCE : 注解只存在于源码中(.java文件),被编译成.class文件后,注解会被忽略.
  2. RetentionPolicy.CLASS : 注解不仅存在于源码中,还存在于.class文件中,但是当被JVM加载到内存中时,注解会被忽略.(也就是在程序运行时,获取不到注解了)
  3. RetentionPolicy.RUNTIME : 注解会存在于源码、.class文件和运行期间,就是当我们在程序运行时,可以获取到该注解.

@Target

@Target主要定义了注解的可以定义哪些元素,我们可以看该注解的源码:

该注解中有个ElementType类型的数组,说明我们可以在修饰注解时, Target里的值可以定义多个,ElementType的类型有:

  1. ElementType.ANNOTATION_TYPE
  2. ElementType.CONSTRUCTOR
  3. ElementType.FIELD
  4. ElementType.LOCAL_VARIABLE
  5. ElementType.METHOD
  6. ElementType.PACKAGE
  7. ElementType.PARAMETER
  8. ElementType.TYPE
  9. ElementType.TYPE_PARAMETER
  10. ElementType.TYPE_USE

具体含义参照JDK源码注释.

@Inherited

@Inherited注解表明注解是否会被子类继承.

@Documented

@Documented注解表明制作javadoc时,将注解信息加入文档.如果注解在声明时使用了@Documented,则在制作javadoc时注解信息会加入javadoc.
